## 0. 三种部署形态 TL;DR

| 形态 | 用途 | 启动命令 | 客户端如何连 |
|---|---|---|---|
| **本地 dev**（go run） | 单人改代码、跑 unit / smoke | `cd server && make run-dev`（仅 gateway）<br>或 `make run-all-dev`（三服务聚合） | `imGatewayHttp = http://<开发机 IP>:8080` |
| **本地 docker-compose** | 起 PG + Redis + Pulsar + OTel + Jaeger 跑集成测试 | 仓库根 `docker compose up -d` | 仅依赖；gateway 仍走 `make run-dev` |
| **pre 集群**（K8s `im-v2`） | 联调 / 压测 / 客户端真接 | `source scripts/pre-env.sh && ./scripts/v4-prepare.sh && kubectl apply -f deploy/k8s/rendered/` | cses Java 下发 `imGatewayHttp = http://192.168.6.41:30880` |

> **生产环境**当前不存在（本项目 `docs/GOAL.md` 阶段还在 pre 验证）。下面所有「集群」操作都指 `pre` (`im-v2` ns)。

### 0.1 服务全景（必看，否则下面命令会让人困惑）

im 后端只跑 **2 个真实服务进程**，外加 1 个**已实现但未拆分**的功能模块、1 个**预留可拆分入口**：

| 服务 / 模块 | 入口 / 位置 | 端口 | 形态 | 当前状态 |
|---|---|---|---|---|
| **gateway** | `cmd/gateway` | HTTP/WS `:8080` | 业务入口（cses-client 唯一接入点）| ✅ 生产就绪，独立进程 |
| **message** | `cmd/message` | 无 HTTP，Pulsar consumer | 消费 `incoming-message` topic → 落库 + 触发推送 | ✅ 生产就绪，独立进程（旁路模型）|
| **sync**（业务功能）| `internal/http/sync.go` + `internal/service/sync.go` | 走 gateway 的 `:8080` | gateway 进程内的 `POST /api/sync` handler，**和 gateway 同生共死** | ✅ 生产就绪，**不是独立进程** |
| `cmd/sync`（预留入口）| `cmd/sync/main.go` 16 行 | — | **空壳 main**，给"未来把 sync 拆成独立 binary 部署"预留 | ⚠️ 跑起来立即退出，**当前阶段不需要也不应该单独启** |

**重要澄清**：

> **sync 业务**早就实现了。`POST /api/sync` 是 cses-client 增量拉历史的核心 endpoint，96 行 handler + 87 行 service 都在 `internal/`，cutover 阶段 mirror 了 cses Java legacy `SyncRequest` 的 wire shape。**你只要 `make run-dev` 起 gateway，sync 业务就在跑**——`curl -X POST http://localhost:8080/api/sync -H 'CookieId: xxx' -d '{"channels":[]}'` 即可验证。

> **`cmd/sync` 是预留可分离部署的占位**：当前 sync 跟 gateway 共享内存里的 channelRepo / messageRepo / log，所以打包在 gateway 进程内最经济（少一次跨进程调用）。`cmd/sync/main.go` 只有 16 行 `printf + return 0`，是给"将来 sync 流量大到要单独水平扩展时"的入口骨架。**现在跑它会立刻退出，这是设计，不是 bug**。

> **为什么 message 单独拆**：M2 设计把"写入消息 → 推送"做成 Pulsar 异步管道（gateway publish → message consume → 落库 + push），这一段不需要共享内存上下文，拆开后能独立扩缩容。但本地开发也可以**不起 message**——gateway 进程内有同样的消费逻辑兜底，仅当你要测「跨进程消息流」或者模拟生产部署时才单独起 message。

## 1. 本地 dev：单人改代码

### 1.1 前置依赖

- Go 1.26+（`go.mod` 锁的版本）
- Docker（起 PG / Redis / Pulsar 依赖）
- `gotestsum`（`make test` 用），`golangci-lint`（`make lint` 用）
- `kubectl`（仅当连 pre 集群时需要）

### 1.2 启动依赖（本地 docker-compose）

```bash
# 仓库根目录
docker compose up -d           # postgres:15432 / redis:16379 / pulsar:6650
docker compose ps              # 5 个容器应该都 healthy
```

容器与宿主机端口（**不要改这些端口号**，`server/config.example.yaml` 已硬编码）：

| 服务 | 容器名 | 宿主端口 | 用途 |
|---|---|---|---|
| postgres | im-postgres | `15432` | im 主库 |
| redis | im-redis | `16379` | routing / presence |
| pulsar | im-pulsar | `6650`（broker）/ `8088`（admin） | 跨 pod 推送 |
| otel-collector | im-otel-collector | `4317`（gRPC）/ `4318`（HTTP）| trace 收集 |
| jaeger | im-jaeger | `16686`（UI） | trace 查看 |

### 1.3 跑数据库 migrations（首次或 schema 变更后必跑）

```bash
cd server
DATABASE_URL='postgres://im:im@localhost:15432/im?sslmode=disable' make migrate-up
```

### 1.4 编译（可选，跑 dev 不一定要先编译）

```bash
cd server

# 单服务编译（产物在 server/bin/{gateway,message,sync}）
make build-gateway
make build-message
make build-sync

# 一键全编译
make build-all

# 跑代码静态检查
make verify-build      # = go build ./... + go vet + golangci-lint
```

> `make run-dev` 系列用的是 `go run`，会自动编译并跑临时二进制，**不必先 make build**。`make build-*` 主要给"想跑 ./bin/gateway 命令行调试"或者 CI 用。

### 1.5 启动 —— 6 种 dev 形态

> **重要前提**：sync 业务（`POST /api/sync`）跑在 gateway 进程内，不是独立服务。所以只要起了 gateway，sync 端点就能用。`cmd/sync` 是预留可拆分入口，**默认不需要启**。

`server/Makefile` 内置 6 条 dev target，按"真实运行 vs 预留入口"分两组：

**真实服务 target**（生产里也跑的进程）：

| Target | 启的服务 | 形态 | 用途 |
|---|---|---|---|
| `make run-dev` | gateway（含 sync 业务）| 当前终端前台 | 日常单服务调试（最常用，sync `/api/sync` 在此可用）|
| `make run-message-dev` | message | 当前终端前台 | 单独调 Pulsar 消费链路 |
| **`make run-all-dev`** | **gateway + message** | **单窗口聚合 + 彩色前缀** | **一键起全栈两个真实进程** |
| **`make run-all-dev-windows`** | **gateway + message** | **tmux 2 窗口 / Terminal 2 窗口** | **长时间盯单服务日志** |

**预留入口 target**（开发演示用）：

| Target | 行为 | 何时用 |
|---|---|---|
| `make run-sync-stub` | 跑 `cmd/sync` 16 行预留入口，立即退出 | 仅验证 `cmd/sync` binary 可编译 |
| `make run-all-dev-with-sync` | gateway + message + sync stub（多一行立即退出的输出）| 演示"未来 sync 拆分形态" |
| `make run-all-dev-windows-with-sync` | tmux 3 窗口含 sync stub | 同上多窗口版 |

下面分别说明。

#### A. 单服务模式：`make run-dev`（最常用）

```bash
cd server && make run-dev
```

只起 gateway，等价于 `go run ./cmd/gateway`。当前终端前台跑，Ctrl+C 停。
启动成功的标志：

```
gateway listening on :8080  gateway_id=<HOSTNAME-或-UUID>
```

#### B. pre-tunnel 模式：`make run-pre-tunnel`（高危，复现 pre bug 用）

```bash
cd server && make run-pre-tunnel
```

> **危险**：会写入和 pre 集群同一个 `im_pre` 库。复现完 bug **必须立刻停掉**，不要长时间挂着。

#### C. 全栈聚合模式：`make run-all-dev` ⭐

```bash
cd server && make run-all-dev
```

**效果**：单窗口同时跑 gateway + message 两个真实进程，每行输出带 ANSI 彩色前缀区分：

```
[gateway] gateway listening on :8080 ...
[message] consumer subscribed to im_dev/incoming-message ...
[gateway] hub.register user=u123 conn=...
[message] cross_pod_push.publish topic=...
[gateway] sync handler /api/sync hit user=u123 channels=3   ← sync 业务在 gateway 里
```

- 前缀颜色：gateway=绿，message=青
- 同时落本地日志：`server/logs/dev/{gateway,message}.log`（已加入 `.gitignore`）
- **Ctrl+C 一次性结束所有子进程**（脚本 trap 信号 + 杀进程组，不会留孤儿 `go run`）
- 适用场景：日常全栈开发、看整条消息链路的时序关系

> **要演示"未来 sync 拆分"形态**：`make run-all-dev-with-sync` 会额外起 sync stub，多一行黄色的 `[sync] sync service starting...` 然后立刻消失。**注意 sync stub 退出不会影响 gateway / message**，脚本设计了不互拖累。

#### D. 全栈多窗口模式：`make run-all-dev-windows` ⭐

```bash
cd server && make run-all-dev-windows
```

**效果**：开 2 个独立窗口分别跑 gateway / message，方便长时间盯单个服务的日志。

优先 **tmux session**（如果装了 tmux）：

```bash
tmux attach -t im-dev          # 进入 session
# Ctrl+B 0/1   在 gateway/message 两个 window 之间切换
# Ctrl+B d                       从 session detach（不停服务）
tmux kill-session -t im-dev    # 整体停掉
```

没装 tmux 时**自动回退到 macOS Terminal.app**，通过 `osascript` 开 2 个独立窗口；关闭对应窗口或 Ctrl+C 即可停。

> 装 tmux 推荐：`brew install tmux`。tmux 模式比 Terminal.app 模式更可控（不会窗口爆炸、可远程 attach）。
>
> 加 sync stub 第 3 个窗口：`make run-all-dev-windows-with-sync`。

#### 模式选择建议

| 我的情景 | 选哪个 |
|---|---|
| 只改 gateway HTTP/WS handler（含 sync `/api/sync`）| `make run-dev` |
| 改了 message 消费逻辑 / Pulsar topic 流 | `make run-message-dev`（单独）或 `make run-all-dev`（看整链）|
| 跑 client smoke / k6 压测 / 联调 cses-client | `make run-all-dev`（聚合好看时序）|
| 调一个偶发的 race / 死锁，需要长时间盯日志 | `make run-all-dev-windows`（tmux）|
| 演示 / 验证未来 sync 拆分入口 | `make run-all-dev-with-sync` |
| 跑 unit test / 集成测试 | 不需要起 dev 服务，直接 `make verify-unit` / `verify-integration`（testcontainer 自起依赖）|

## 2. pre 集群（K8s `im-v2`）：联调 / 压测 / 客户端真接

### 2.1 谁负责按钮：用户人工触发

按 `SESSION.md §0` 当前 backlog 约定，**pre 部署 `v1.0.0-pre-7h` 等用户人工触发**。Claude / agent 不主动 `docker push` 或 `kubectl apply`，除非用户在当前会话明确说「上 pre」。

### 2.2 构建 + 推镜像 + 渲染 + 应用（全流程）

仓库根目录：

```bash
# Step 1 — 加载 pre 集群默认参数（PG / Redis / Pulsar SVC DNS）
source scripts/pre-env.sh

# 默认参数（pre-env.sh 内置，必要时 override 在 source 之后 export 覆盖）：
#   NAMESPACE=im-v2
#   REGISTRY=harbor.jinqidongli.com/x9-go/im
#   IMAGE_TAG=v1.0.0-pre   # 想换 tag 就 export IMAGE_TAG=v1.0.0-pre-7h 再 source
#   PG_DSN=postgres://...@postgresql-cses-pre-cnpg-rw.postgres-cses.svc:5432/im_pre
#   REDIS_ADDR=redis-cses-pre-redis-cluster-headless.redis-cses.svc:6379
#   REDIS_CLUSTER=true
#   PULSAR_URL=pulsar://pulsar-cses-broker.pulsar-cses.svc:6650
#   JWT_SECRET=<openssl rand -hex 32>  # 每次 source 重新生成，除非 pre-export

# Step 2 — 构建 + 推镜像 + 渲染 manifest 到 deploy/k8s/rendered/
./scripts/v4-prepare.sh
# 跳过构建：SKIP_BUILD=1 ./scripts/v4-prepare.sh
# 跳过推送：SKIP_PUSH=1 ./scripts/v4-prepare.sh

# Step 3 — 应用到集群
kubectl apply -f deploy/k8s/rendered/

# Step 4 — 等 rollout（3 副本全就绪 ≤ 120s）
kubectl -n im-v2 rollout status deploy/im-gateway --timeout=120s
```

### 2.4 重新构建（改完代码后只想热替换镜像）

最常见场景：只改了 Go 代码，PG / Redis / Pulsar / Consul 配置都没动。

```bash
# 在 server 仓库根
source scripts/pre-env.sh
export IMAGE_TAG=v1.0.0-pre-$(date +%H%M)   # 新 tag 避免 imagePullPolicy=IfNotPresent 命中旧缓存
./scripts/v4-prepare.sh

# 渲染过的 deployment 会带上新 tag，apply 后触发 RollingUpdate
kubectl apply -f deploy/k8s/rendered/20-deployment.yaml
kubectl -n im-v2 rollout status deploy/im-gateway --timeout=120s
```

> **为什么必须换 tag**：`20-deployment.yaml` 的 `imagePullPolicy: IfNotPresent`，同 tag 不会重新 pull。强行用同 tag 必须 `kubectl rollout restart deploy/im-gateway` 才能拉新镜像，但只有当镜像是 mutable repo 时才行（pre 的 harbor 通常关闭了 mutable）。

### 2.5 停止 / 回滚

```bash
# 回滚到上一个版本（保留 Deployment）
kubectl -n im-v2 rollout undo deploy/im-gateway

# 全量下线（删 Deployment / Service / HPA / PDB / ConfigMap / Secret，PG 数据保留）
kubectl delete -f deploy/k8s/rendered/

# 临时缩容到 0（保留所有 manifest）
kubectl -n im-v2 scale deploy/im-gateway --replicas=0
```

> **不要直接删 namespace**。`im-v2` 是共享 ns，里面还有未来的 message / sync 服务以及 cses Java 侧的关联资源。

### 2.6 单 pod 优雅重启（不影响在线 WS）

```bash
kubectl -n im-v2 delete pod -l app=im-gateway --field-selector=status.phase=Running --grace-period=30
```

- `terminationGracePeriodSeconds: 30` + `preStop sleep 10` 给 Service 摘流时间（`20-deployment.yaml`）。
- 客户端会收到 `1001 GoingAway`，30s 内自动重连到另一个 pod（M2 reconnect 逻辑）。
- PDB `minAvailable=2` 保证不会一次性挂三个。

## 3. 暴露给客户端：cses-client 怎么连上来

### 3.1 网络路径

cses-client 是 Angular + Tauri，**跑在用户的 Mac/Windows 上，不在 K8s 集群内**，所以集群 SVC DNS（`im-gateway.im-v2.svc.cluster.local`）对它**完全不可达**。

唯一暴露通道是 **NodePort 30880**（`deploy/k8s/30-service.yaml`）：

```
cses-client (用户机)
    │
    ▼
http://192.168.6.{41,42,43}:30880   ← 集群任一 worker 的 IP
    │
    ▼ kube-proxy iptables
im-gateway pods (3 副本，:8080)
```

> **不要把这个 Service 改回 ClusterIP**。`30-service.yaml` 顶部有大段注释解释踩过的坑：webview HTTP/WS 会 hang 在 `status=0 / "Handshake not finished"`。NodePort 30880 是和客户端约定好的硬编码（在 cses Java 下发字段里也写死了）。

### 3.2 客户端如何拿到 gateway 地址：cses Java 下发

cses-client **不**硬编码 gateway 地址。流程：

1. 用户在 cses-client 登录 → 客户端调 **cses Java**（老栈）的 `/login`。
2. cses Java 在登录响应 / 配置接口里下发：

   ```json
   {
     "imGatewayHttp": "http://192.168.6.41:30880",
     "imGatewayWebSocket": "ws://192.168.6.41:30880",
     "imGatewayEnabled": true
   }
   ```

3. cses-client 启动时读这三个字段：
   - `imGatewayEnabled: true` + `imGatewayHttp` 非空 → `apiFlavor = "im"`，业务调用切到 im。
   - 否则 → `apiFlavor = "mattermost"`，走 cses Java 老路径。

cses Java 侧控制开关：`XPA-12.yaml` 里的 `imGatewayEnabled: true` + 三个字段值。改这个 YAML 后 cses Java 需要重启。

> 详细字段含义、apiFlavor 切换时机、Console 验证 log 见 `docs/CSES_CLIENT_内部对接契约.md §6`。

### 3.3 鉴权：CookieId 双栈复用

im **不签自己的 JWT 给最终用户**，而是复用 cses Java 已经发的 `CookieId`（HTTP Header）+ Redis 共享会话：

```
客户端 HTTP/WS request
  Header: CookieId: 69eec6dbe6876865ff98945a
  ↓
im gateway middleware
  ↓ redis-cli HGET User '"<cookieId>"' → 拿到 user 资料
  ↓ 未命中 → 401
继续业务
```

> 这就是为什么 dev / pre / prod 三个环境的 `JWT_SECRET` 都没有「正确」一说——它只用于 im 自己内部签发的工具 token（如 sync 拉数据时的服务间凭证），不发给客户端。

### 3.4 客户端连接自检三连

跨网机器（开发机 / 测试机 / 客户端宿主）只要满足这三条，就能确认 im 已经被正确暴露：

```bash
# 1) NodePort 可达
curl -i http://192.168.6.41:30880/healthz
# 期望：HTTP/1.1 200 OK

# 2) 业务路径鉴权链路通
curl -i -H 'CookieId: <真实-cookie-id>' http://192.168.6.41:30880/api/me
# 期望：200 + {"status":"success","data":{"user":{...}}}
# 401  → CookieId 在 cses Redis 里没了（用户被 kick / 过期 / 跨环境）
# 502  → pod 在但 readyz 还没绿（看 §4.1）

# 3) WS upgrade 通
wscat -c 'ws://192.168.6.41:30880/ws?cookieId=<真实-cookie-id>'
# 期望：握手成功，收到 { "type": "connected", "data": {...} } 之类的首帧
```

cses Java 侧最容易漏的一步：**`imGatewayEnabled: true` 改完没重启 cses Java**。结果客户端拿到的还是旧配置，apiFlavor 一直停留在 `mattermost`，看似 im 后端「没起作用」其实根本没被打到。

## 4. 故障排查（按问题归类）

### 4.1 集群侧：pod 起不来 / 起来了 readyz 不绿

**症状判断顺序**：

```bash
kubectl -n im-v2 get pods -l app=im-gateway
```

| READY 列 | 含义 | 下一步 |
|---|---|---|
| `0/1 ImagePullBackOff` | 镜像没推到 harbor / tag 拼错 | `kubectl describe pod` 看 `Failed to pull image`，确认 harbor 有这个 tag |
| `0/1 CrashLoopBackOff` | 容器启动后立刻退出 | `kubectl logs <pod> --previous` 看上一次崩的原因 |
| `0/1 Running` + 久不就绪 | `/readyz` 一直 5xx | 见下文 |
| `1/1 Running` 但客户端连不上 | NodePort 链路问题 | 见 §4.2 |

`/readyz` 不绿的常见原因（按命中率排序）：

1. **PG 连不上**：
   - `kubectl logs <pod> | grep 'pg.*connect'`
   - 检查 ConfigMap `im-gateway-config` 里 `pg.dsn` 是不是 pre-env.sh 那条 SVC DNS。
   - 检查 pre 集群里 `postgresql-cses-pre-cnpg-rw` 这个 SVC 是不是健康：`kubectl -n postgres-cses get svc | grep cnpg-rw`。

2. **Redis 连不上 / 鉴权全 401**（Cluster 模式 vs single 模式陷阱，**最易踩**）：
   - 现象 A：`/readyz` 一直 5xx，gateway 日志 `redis.*timeout` / `MOVED` 错误
   - 现象 B：`/readyz` 绿但所有业务接口返回 `401 missing auth: cookieId header required`，即便 cookie 真的在 Redis 里能 `GET UserData:<id>` 拿到
   - **根因**：consul-pre 下发的 `im-go/dev/config.yaml` 配 `cluster: false`，但 pre Redis 实际是 Cluster 部署。go-redis 跑 single client 时连 Cluster headless DNS 只能拿到一个 slot 的 master，跨 slot 的 key（`UserData:<id>` 跟 routing 在不同 slot）就拿不到。
   - **修复**：必须 env override `IM_REDIS_CLUSTER=true`。
     - **本地 dev**：`server/Makefile` 的 `run-dev` / `run-message-dev` / `run-pre-tunnel` / `run-sync-stub` **已强制写入** `IM_REDIS_CLUSTER=true`（2026-05-12 落地），`run-all-dev.sh` 的 `COMMON_ENV` 也同步。直接 `make run-dev` 即可，不必手动 export。
     - **pre 集群**：`20-deployment.yaml` 里 `__REDIS_CLUSTER__` 占位由 `pre-env.sh` 的 `REDIS_CLUSTER=true` + `v4-prepare.sh` 替换成 `true`。
   - **如果你绕过 Makefile 直接 `go run ./cmd/gateway`**：必须自己 export `IM_REDIS_CLUSTER=true`，否则 100% 踩坑。
   - 详见 `docs/harness/C010 §4.6` + `docs/apifox-sync/01-auth-and-sync.md §6 troubleshoot 表`。

3. **Pulsar 连不上**：
   - `kubectl logs <pod> | grep -i pulsar`
   - 检查 `PULSAR_URL` 是否指向 `pulsar-cses-broker`（broker）不是 `pulsar-cses-proxy`（proxy 走 32650 NodePort，仅外部访问用）。
   - **本地 dev 多人共用 pre Pulsar** → 必须按 `harness/C003` 给 topic 加 `.{HOSTNAME}` 后缀，否则消息会被别人的本地 gateway 抢走。

4. **OTel collector 连不上**：
   - 不阻塞 readyz；只是 trace 上报失败，gateway 还能正常服务。
   - `OTEL_DISABLED=true` 可以彻底关。

### 4.2 客户端侧：cses-client 连不上 im

**第 0 步：确认 apiFlavor 切了没**

打开 cses-client devtools Console，搜：

```
🔀 [MessageHttpService] apiFlavor: mattermost → im
```

- 没看到 → cses Java 没下发 `imGatewayHttp`：检查 cses Java 侧的 XPA-12.yaml `imGatewayEnabled: true`，**改完必须重启 cses Java**。
- 看到了但接着是 `mattermost → mattermost`（没切）：`imGatewayHttp` 字段空或 null，看 cses Java 登录响应原始 payload。

**第 1 步：HTTP 通不通**

```bash
curl -i http://192.168.6.41:30880/healthz
```

- 拒绝连接 → 集群侧 NodePort 没开，看 `kubectl get svc im-gateway`，`PORT(S)` 列必须是 `8080:30880/TCP`。
- 超时 → 防火墙 / VPN：客户端到 192.168.6.x 段是否可达。
- 200 但客户端还是连不上 → 看 §4.3 WS 单独排查。

**第 2 步：业务接口 401**

```bash
curl -i -H 'CookieId: <从客户端 devtools 复制>' http://192.168.6.41:30880/api/me
```

- 401 → `redis-cli -h <pre-redis> HGET User '"<cookieId>"'`，没值就是 cses Java Redis 里的会话已经清了。让用户重新登录。
- 200 但返回的 user_id 跟期望不符 → 双栈会话错乱，清 cses-client 本地 SurrealDB 缓存重登。

**第 3 步：响应 shape 不对**

im 现在走全局信封（`harness/C007`）：

```json
{"status":"success","data":{...}}      // 成功
{"status":"error","error":"<message>"} // 失败
```

如果客户端拿到的是 raw 数据没 envelope → handler 走了 `harness/C007` 黑名单（双重 wrap 或绕过中间件）。grep 仓库：

```bash
grep -rn '"status":\s*"success"' server/internal/http/handlers/ | grep -v _test.go
```

正常情况下 handler 里不应该出现这一行（中间件统一加）。

### 4.3 WS 专项：upgrade 401 / 立刻断开 / 收不到推送

| 症状 | 原因 | 排查 |
|---|---|---|
| WS upgrade 返回 401 | CookieId 没在 cses Redis | `redis-cli -h <pre-redis> HGET User '"<cookieId>"'` |
| WS 握手成功但 1s 内断 | gateway 把它当僵尸（routing 写失败）| 看 pod 日志 `hub.register` 后是否紧跟 `hub.unregister`，看 Redis 是否 `SET im-new:routing:<userId>` 成功 |
| A 发消息 B 收不到（同 pod）| hub 推送丢 | `im_push_local_success_total` 计数没涨 → 看 `hub.go` 单测；涨了但客户端没收 → WS 心跳卡死，看 `harness/C004` 心跳 15s × 3 |
| A 发消息 B 收不到（跨 pod）| Pulsar fan-out 没起 | `im_push_cross_pod_success_total` 必须涨；不涨看 `cross_pod_push.publish` 日志；走 `harness/C002` `gateway.CrossPodPush` 路径检查 |
| 收到的 type 客户端不认识 | 22 type 之外的新 type 漏了客户端正规化 | 看 `harness/C005` 22 type 清单，新增必须走 V2 RFC |

### 4.4 集群 P0：所有 pod 502 / readyz 全红

按下面顺序快速降级：

```bash
# 1. 看是不是 PG / Redis / Pulsar 中间件挂了
kubectl -n postgres-cses get pods | grep cnpg
kubectl -n redis-cses    get pods | grep redis-cluster
kubectl -n pulsar-cses   get pods | grep broker

# 2. 中间件好的话，回滚 gateway 到上一个稳定版本
kubectl -n im-v2 rollout undo deploy/im-gateway

# 3. 实在不行，缩容到 0 让 cses-client 自动 fallback 到 apiFlavor=mattermost
#    （仅当 cses Java 同时把 imGatewayEnabled 翻成 false 才有效，否则客户端会一直试 im）
kubectl -n im-v2 scale deploy/im-gateway --replicas=0
```
