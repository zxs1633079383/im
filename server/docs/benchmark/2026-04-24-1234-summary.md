## 瓶颈定位

### CPU 没满 → 不是 CPU 瓶颈
采样时 `kubectl top pods` 显示 gateway CPU 总和 < 5% of cluster limit（soak 末段 ramp-down，瞬时值低）。说明**扩出来的 pod 不是 CPU 饱和驱动的**，HPA 是有效扩容但 CPU 使用率误导。

### 真瓶颈：HTTP 连接建立 & PG 连接池
- VU=100 起就出现长尾：p50 6ms（正常），p95 1.46s，p99 几秒
- http_req_failed 36.9% @ VU=100，69% @ VU=300 — 大量 HTTP 请求超时或拒绝
- **im_action_ok 98%** @ VU=100 说明**业务逻辑一旦进入 handler 就能正确完成**
- 结论：**瓶颈在 "HTTP 请求从 k6 → gateway → PG/Redis" 管道建立阶段**，而非 gateway 业务代码

### 可能原因
1. **PG 连接池**：`pg.max_conns=20` per pod × 3 pods = 60 连接；200+ 并发 VU 在 login/DM create 阶段挤压
2. **每 VU 双登录**：fullchain VU 同时 login 自己和 peer（两次 bcrypt），300 VU 起就 600 次并发 bcrypt
3. **WS upgrade 不 reuse HTTP 连接**：每 VU 独立 TCP + WS handshake，k8s svc LB + kube-proxy 转发有一定并发上限
4. **PEER_POOL=50** 意味着 50+ VU 同时 login 同一 peer user → PG 行锁 + bcrypt

## 下轮建议（下一个 loop 前）

1. **修 `im_push_e2e_ms` 埋点**：改用 server 端 `created_at` 字段或在 message content 嵌 timestamp，让接收 VU 也能算端到端延迟
2. **减少 VU 启动开销**：seed 时建一个 **共享 peer 账号**（比如 `k6peer`），所有 VU 只 login 自己 + 读取 peer_id 缓存；省一半 bcrypt
3. **加 PG 连接池大小**：`pg.max_conns` 20 → 40 per pod；或加 pgbouncer
4. **HPA 下限调整**：`minReplicas: 3 → 6` 或 7，避免起始 3 pod 在 ramp 早期被打爆
5. **benchmark-loop.sh threshold 增加**：`action_ok < 0.95` 或 `http_req_failed > 10%` 也算 stop

## 真实生产推断

- 当前 pre 3-pod baseline 承受能力 ≈ **稳态 100-150 并发 WS + 10-20 msg/s**
- 以 HPA 11 pod 峰值推算，**单 pod 稳态承受 ≈ 130 VU**（VU=1500 时 11 pod = 136/pod）
- 要达到 BACKEND.md §5.5 S5 目标的 **150k VU + 10k msg/s**，按此推断需要约 **1000+ pod** 或调优每 pod 到 1-2k 连接级（目前 k8s 业界标杆）
- **结论**：架构正确，**需要单 pod 性能调优**（Go GOMAXPROCS / 连接池 / bcrypt 替换为 scrypt/argon2 / 数据库连接复用）才能达到高并发 SLO
