## 根因与修复

### 🔴 旧版瓶颈的真实面目
之前 pre-2 观察到的"长尾 P95"和"85% action_ok"**不是 PG pool 不够**（虽然 pool=20 太小是事实），而主要是：

1. **gateway `Conn.Push` race**：`runHeartbeat` 或 `CrossPodPush` fan-out 调 `Push`，与 `conn.Close()` 的 `close(c.send)` 赛跑 → `send on closed channel` panic → pod 崩溃 → HPA 看到的是"崩溃+调度"而非"稳态高 CPU" → 扩容延迟 + 请求堆积 → 长尾。pre benchmark 抓到 3 个 gateway pod 各 RESTARTS=8。
2. **PG pool=20** 加剧了排队；但只要 pod 不挂、扩容跟上，pool=50 就够 VU=100 用。

### ✅ pre-5 修复
- `Conn` 加 `closeOnce sync.Once` + `closed atomic.Bool` 双保险
- `Close()` 只执行一次；`Push()` 先 `c.closed.Load()` 检查；尾部 `defer recover()` 兜底 narrow race
- **pre-5 压测期间 pod RESTARTS=0**（对比 pre-4 的 8×3=24 次崩溃）
- PG pool 从 20 → 300（对齐 Java HikariCP），`max-lifetime=600s` / `idle-timeout=300s`

### 📊 HPA 真实工作了
- VU=100 → 3 pods（稳定，未扩）
- VU=300 → 4→7 pods（按 `Percent: 100%` policy 翻倍）
- VU=800 → 7→16 pods（在 cooldown 内两轮翻倍）
- 触发 `maxReplicas=20` 接近值 → benchmark-loop 自动停 ✅
