## 根因与修复

### 旧 FindDM SQL
```sql
SELECT c.* FROM channels c
JOIN channel_members ma ON ma.channel_id = c.id AND ma.user_id = $1
JOIN channel_members mb ON mb.channel_id = c.id AND mb.user_id = $2
WHERE c.type = 1 LIMIT 1
```
PG planner 在并发下倾向 bitmap scan `idx_channel_members_channel` + re-filter `user_id` → mb 端整个 channel 成员都要扫，大群放大。

### 新 FindDM SQL（EXISTS 驱动）
```sql
SELECT c.* FROM channel_members ma
JOIN channels c ON c.id = ma.channel_id AND c.type = 1
WHERE ma.user_id = $1
  AND EXISTS (
    SELECT 1 FROM channel_members mb
    WHERE mb.channel_id = ma.channel_id AND mb.user_id = $2
  )
LIMIT 1
```

### 新 index
```sql
CREATE INDEX idx_channel_members_channel_user ON channel_members(channel_id, user_id);
```
PK `(user_id, channel_id)` 的"反向"形式。EXISTS 探测按 `(channel_id, user_id)` 走 **index-only scan**。

### 执行路径
1. `ma.user_id = A` → PK 索引，拿 A 的所有 channel_id（通常 ≤ 几十）
2. 每个 channel JOIN c.type=1 过滤 → PK 查
3. EXISTS mb.channel_id=X AND mb.user_id=B → **新 index index-only**
- 单次 DM lookup：**< 1 ms** on warm cache（旧版 bitmap scan 3000 ms）

## 为什么 sync/markRead 也快了

虽然 DM 查询的直接调用点在 `CreateOrGetDM` handler（VU 启动时只调一次），但在旧版 pre-5 每次 VU 在 ramp 阶段 DM lookup 3s 会：
1. 占用 PG 连接池（300 VU × 1 conn × 3s = 15 min 连接×秒）
2. 排队阻塞其他请求（sync/markRead 在同 pod 等 PG 连接）
3. 打 gorm slow-sql 日志影响 I/O

index 生效后 PG 连接即用即还，其他 SQL（sync/markRead 的 join）无排队，P95 自然降下来。
