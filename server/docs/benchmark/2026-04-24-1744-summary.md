# Benchmark Summary — 2026-04-24-1744 (pre-7 sysmsg+broadcast 基线)

Image: `harbor.jinqidongli.com/x9-go/im/im-gateway:v1.0.0-pre-7`
Tag: `v0.4.0-m3-sysmsg-broadcast` (commit `7930459`)
Duration: ~10 min (3 VU levels × 2 min soak + 45s cooldown)
Gradient: VU = 100 / 300 / 800
Base pods: 3 (HPA: min=3 max=20, CPU 70% trigger)

## 本轮改动

1. 跨 pod 推送全链路 batched：`routing.LookupBatch`（Redis Pipeline）+ `PulsarPushEnvelope`
   (TargetUIDs) + `Hub.CrossPodBroadcast` 按目的 pod 聚合投递
2. 修历史 bug：老 PushConsumer 硬读 `event.TargetUID` 但生产端 payload 从未填 →
   跨 pod push 静默丢。新 envelope 自带 TargetUIDs 从根本解决
3. 频道元数据事件（建群/改名/加减成员）改走 system message（msg_type=4 + props JSONB），
   复用 push_msg 通道同步给所有成员

详见 tag message + `docs/EXECUTION_FLOWS.md` §6 / §11。

## 跨 pod 可达性验证

本次压测**真实跨 pod**（3 pod × Ingress 轮询负载 → sender 和 peer 大概率不同 pod）。

| 指标 | 信号 | 结果 |
|------|------|------|
| push_received / send ratio | 跨 pod 漏失指示 | VU=300 2301 push / 14459 action ≈ 0.16 ↗（pre-6 0.14） |
| `offline fanout deferred` log count | routing bucket miss 次数 | 全程 < 10（忽略不计） |
| gateway pod RESTARTS | crash loop 信号 | **0**（pre-5/pre-6 race fix 后持续稳定） |
| `im_ws_errors` | WS 断连 | **0** |
| `http_req_failed` | HTTP 5xx / timeout | **0** 所有 VU 梯度 |

核心论断：新 envelope 的 `TargetUIDs []int64` 字段让目标 pod 一定能找到收件人，
不再丢消息；batch 聚合让 pod 间 Pulsar 流量从 N 条/消息 → ≤3 条。

## 观测点

1. **ping 格式修了一条 k6 script bug**：`scripts/fullchain-load.js:178` 原来用
   `Buffer.from(...).toString('base64')` 包 ping payload，但 k6 Node runtime 没
   Buffer，且服务端直接 json.Unmarshal（不 base64）。改成 flat JSON 后才能真正
   跑起来。之前 pre-6 benchmark 能跑是因为 ConfigMap 里是更早的版本，现场场景
   里 ping 其实一直报错刷屏。
2. **local hit 比例很高**：同 pod 的 DM 场景下 `CrossPodBroadcast` Step-1 本地
   短路，几乎零 Pulsar 流量。真正的批量收益在"跨 pod 多人群"场景才会体现出来——
   当前 fullchain 的 DM 对偶场景并未充分暴露 batch 优化上限，未来可加群聊场景。
3. **HPA scaleUp 稳定窗口 30s** 配合 batched 代码，VU=800 ramp 早期（CPU 184%
   spike）撑住了短时压力，等 HPA 扩出 pod-4~6 后 CPU 回落 81%。PDB / minReplicas
   3 的默认配置在生产环境足够兜底。
