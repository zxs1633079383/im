# v0.6.3 — WS cookie 化 测试计划

**目标**：把 message-v3 / cses-client 的主 WebSocket 从 `cses-server (/api/v4/websocket)` 切到 `im-gateway (/ws)`，鉴权由 cookieId 直接驱动（不再 mint JWT）。

**接受门**：所有用例 ≥ 5/5 PASS（Header / Query / 无 cookie 401 / push_msg 端到端 / e2e-pre-m4 HTTP 回归）。

---

## 1. 服务侧改动（已 commit `d36b0e1`）

| 文件 | 改动 |
|---|---|
| `internal/middleware/mattermost_cookie.go` | export `ResolveCookieID(ctx, rdb, cookieID, log)` 复用 LRU + metrics |
| `internal/gateway/ws_handler.go` | `WithCookieAuth(rdb)` + `authenticate(r)` 三优先级解析（Header → query → JWT fallback） |

## 2. 客户端改动

| 文件 | 改动 |
|---|---|

灰度切换闸门：后端 `auth.imGatewayWs` 有值 → 切到 im-gateway；空值 → 维持 cses 路径。

## 4. 已知遗留 → v0.6.4

- **后端 `imGatewayWs` 配置下发**：M4 login 接口暂未把 `imGatewayWs` 字段加进 auth 响应。临时方案：手动写 `auth_extra` 配置，或在 cses-server 端 mock；正式上线前补。
- **WS 重连时 cookie 已过期**：`reconnectMainWs` 失败时应触发上层重新拉 auth；当前依赖 cookie LRU 30s TTL 自然过期 + 上游 cses 心跳。
- **WS Pulsar fan-out 在 cookie auth 路径上的覆盖率**：现 ws-smoke 覆盖了同 pod 内 alice→bob，但跨 pod 还得靠 fullchain-load 跑一遍。
- **e2e-teardown 适配**：v0.6.2 的 e2e-teardown.sh 还是跟 cses-server 共享数据库的旧脚本，需要新做一份只清 im_pre 的 lite 版本。
